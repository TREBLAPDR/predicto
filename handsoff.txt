📱 PROJECT OVERVIEW
What It Does
An AI-powered shopping list mobile app (Android) that:
* Scans receipts using camera and extracts items with OCR + AI
* Predicts what you need to buy based on purchase history
* Suggests items based on day-of-week patterns and product associations
* Manages products database with PostgreSQL
* Shares lists with family/friends
* Tracks purchase history for smart recommendations
Target Users
Filipino shoppers who want to:
* Save time creating shopping lists
* Never forget items they regularly buy
* Track spending and purchase patterns
* Share lists with family members
Unique Selling Points
1. Dual OCR System: Fast local ML Kit + accurate cloud Gemini Pro
2. Smart Predictions: AI learns your shopping patterns (what you buy and when)
3. Day-Based Intelligence: Knows you buy milk every Monday
4. Product Associations: Suggests items often bought together
5. Philippine Peso (₱): Fully localized for PH market

🛠 TECH STACK
Frontend (Mobile App)
* Framework: Flutter 3.0+
* Language: Dart
* State Management: setState (simple, no external packages)
* Local Storage: SharedPreferences
* HTTP Client: http package
* OCR: Google ML Kit Text Recognition (offline)
* Camera: camera plugin
* Image Processing: image package
* Deep Links: uni_links
Backend (API Server)
* Framework: Python FastAPI
* ASGI Server: Uvicorn
* Database ORM: SQLAlchemy (async)
* Database Driver: asyncpg
* AI Integration: Google Generative AI (Gemini Pro Vision)
* Image Processing: Pillow
Database
* Type: PostgreSQL (Relational)
* Hosting: Render.com (Free Tier)
* Connection: Async via asyncpg
Hosting & Deployment
* Backend: Render.com (Auto-deploy from GitHub)
* Database: Render PostgreSQL (Free - 1GB, 90-day limit)
* Mobile App: APK (not published to Play Store yet)

🗄 DATABASE SCHEMA
Products Table
CREATE TABLE products (
    id VARCHAR PRIMARY KEY,
    name VARCHAR(200) NOT NULL,
    category VARCHAR(50) NOT NULL,
    typical_price FLOAT,
    purchase_count INTEGER DEFAULT 0,
    last_purchased_date TIMESTAMP,
    average_days_between_purchases FLOAT,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    INDEX idx_name_category (name, category),
    INDEX idx_purchase_date (last_purchased_date)
);
Product Associations Table
CREATE TABLE product_associations (
    id SERIAL PRIMARY KEY,
    product_a_id VARCHAR REFERENCES products(id) ON DELETE CASCADE,
    product_b_id VARCHAR REFERENCES products(id) ON DELETE CASCADE,
    co_purchase_count INTEGER DEFAULT 1,
    confidence FLOAT DEFAULT 0.5,
    last_purchased_together TIMESTAMP DEFAULT NOW(),
    INDEX idx_product_pair (product_a_id, product_b_id)
);
Purchase History Table
CREATE TABLE purchase_history (
    id SERIAL PRIMARY KEY,
    product_id VARCHAR REFERENCES products(id) ON DELETE CASCADE,
    purchase_date TIMESTAMP NOT NULL,
    price FLOAT,
    quantity FLOAT DEFAULT 1.0,
    store_name VARCHAR(200),
    INDEX idx_product_date (product_id, purchase_date)
);
Relationships
* products ← purchase_history (One-to-Many)
* products ← product_associations → products (Many-to-Many)

🔑 KEY IMPLEMENTATION DETAILS
1. Gemini Pro Prompt Engineering
Critical prompt instructions for accurate parsing:
* Unit price vs line total distinction (qty > 1)
* Spatial alignment of item names with prices
* Filtering non-product lines (payment info, thank you messages)
* JSON-only output (no markdown, no preamble)
Example prompt excerpt:
CRITICAL PRICE RULES:
1. "price" field must ALWAYS be the UNIT PRICE (price per single item)
2. If you see "2 @ PCK" with total 180.50, the unit price is 90.25 (180.50 ÷ 2)
3. NEVER put the total price in the "price" field when qty > 1
2. Image Preprocessing Pipeline
// Optimal preprocessing for receipts
img.Image? image = img.decodeImage(bytes);
image = img.grayscale(image);           // Remove color noise
image = img.contrast(image, contrast: 140); // Enhance text
image = img.brightness(image, brightness: 10); // Lighten if dark
image = img.copyResize(image, width: 2000); // Optimal OCR size
3. Confidence Scoring Algorithm
Factors (weighted):
* OCR confidence (35%)
* Text density (25%)
* Price patterns (20%)
* Text structure (20%)
Recommendations:
* 80%+: Local only
* 65-80%: Local first, offer online
* 45-65%: Suggest online
* <45%: Require online
4. Day-Based Prediction Logic
// Track which day of week items are bought
for purchase in history:
  if purchase.dayOfWeek == targetDayOfWeek:
    dayPurchases[itemName]++

confidence = dayPurchases[item] / totalPurchases[item]

// Only suggest if bought on this day 2+ times
if dayPurchases >= 2 and confidence >= 0.3:
  suggest(item)
5. Duplicate Detection & Merging
When adding an item:
1. Check if name.toLowerCase() == existing.name.toLowerCase() AND category == existing.category
2. If duplicate found → Show dialog:
o Cancel: Don't add
o Merge: Add quantities together
o Replace: Overwrite existing with new values
6. Unique ID Generation
// Prevent duplicate key errors
final uniqueId = '${DateTime.now().microsecondsSinceEpoch}_$counter';
await Future.delayed(Duration(milliseconds: 2)); // Ensure uniqueness
7. Render.com Cold Start Handling
* Free tier spins down after 15 min inactivity
* First request takes 30-60 seconds to wake up
* Auto wake-up on app launch
* User can manually wake backend
* Extended timeout: 90 seconds for first request

⚠️ KNOWN ISSUES & LIMITATIONS
Current Limitations
1. Render Free Tier Sleep
o Backend sleeps after 15 min inactivity
o First request takes 30-60 seconds
o Workaround: Auto wake-up on app launch + manual wake button
2. PostgreSQL Free Tier Expiry
o Render free databases expire after 90 days
o Need to upgrade to paid plan ($7/mo) or migrate to another host
o Workaround: Backup data before expiry, recreate database
3. Local Storage Only for Lists
o Shopping lists stored in SharedPreferences (local only)
o Not synced across devices
o Future: Add user accounts + cloud sync
4. Limited Offline Mode
o Product predictions require backend
o Sharing requires backend
o Current: Local OCR works offline
5. No User Authentication
o Anyone with backend URL can access products
o Share links have no user restrictions
o Future: Add Firebase Auth or JWT tokens
6. No Real-Time Sync
o Shared lists don't update in real-time
o Manual refresh required
o Future: Add WebSockets or Firebase Realtime Database
Minor Issues
* Image preprocessing can be slow on low-end devices (3-5 sec)
* ML Kit accuracy drops on very dark/blurry images (~60%)
* Gemini occasionally returns prices in string format (handled with fallback)
* Share codes are 8 chars (could be longer for better security)
